
# 산출물 2. 프롬프트 시스템 설계 문서

## 1. 과업 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | 리조트 비수기 활성화 보고서 개요 생성 자동화 |
| 목적 | 한산마리나리조트 비수기 활성화 전략 보고서 개요를 LLM으로 자동 생성 |
| 최종 산출물 | 임원 보고용 전략 개요 초안 |
| 적용 모델 | Claude Sonnet 4.6 |
| 문서 성격 | 모델 선정 결과를 바탕으로 한 실제 운용 프롬프트 설계 문서 |

본 과업의 목적은 자사 정보만 입력하면 비수기 활성화 전략 보고서의 개요를 빠르게 생성하고,
이를 실무자가 후편집하여 임원 보고용 문서로 활용할 수 있도록 하는 것이다.

---

## 2. 문제 정의 및 필요성

한산마리나리조트는 객실 31실 규모의 소규모 리조트이며,
요트 4척과 자체 마리나 시설이라는 차별 자산을 보유하고 있다.
반면 성수기 대비 비수기 가동률이 크게 낮아, 비수기 수요를 높이기 위한 전략 수립이 필요하다.

| 구분 | 현황 |
|---|---|
| 객실 규모 | 31실 |
| 특수 자산 | 요트 4척, 자체 마리나 시설 |
| 타겟 고객 | 30~40대 가족 및 커플 |
| 핵심 문제 | 비수기 평일 예약 저조 |
| 목표 | 비수기 객실 가동률 개선, 패키지 판매 확대 |
| 제약 조건 | 대규모 시설 투자 불가 |

기존에는 보고서 초안을 수작업으로 반복 작성해야 했고,
작성자에 따라 구조와 품질 차이가 발생했다.
따라서 자사 상황을 반영한 구조화된 초안 자동화 시스템이 필요하다.

본 시스템은 단순 아이디어 생성보다 **의사결정용 초안 정리**에 초점을 둔다.

## 4. 모델 검토 및 선정 요약

산출물 1에서 3개 모델을 동일 조건으로 비교하였다.

### 4.1 비교 모델

- Gemini Pro 3.1
- GPT-5.4
- Claude Sonnet 4.6

### 4.2 실험 조건

| 항목 | 내용 |
|---|---|
| 사용 환경 | 코디세이 학습 네이토 API |
| 실험 일자 | 2026-07-05 |
| Temperature | 0.7 |
| Top-p | 1.0 |

| 프롬프트 조건 | 시스템/사용자 프롬프트 동일 |
| 응답 수집 | 모델별 1회 |

### 4.3 종합 점수표

| 순위 | 모델 | 총점 | 평균 |
|:---:|---|:---:|:---:|
| 1위 | **Claude Sonnet 4.6** | **28.00 / 30** | **4.67** |
| 2위 | GPT-5.4 | 27.50 / 30 | 4.58 |
| 3위 | Gemini Pro 3.1 | 27.25 / 30 | 4.54 |

### 4.4 최종 선정 근거

Claude Sonnet 4.6을 최종 선정한 이유는 다음과 같다.

1. 자사 수치와 자산을 가장 적극적으로 반영했다.
2. 전략 방향, 실행 과제, KPI가 가장 자연스럽게 연결되었다.
3. 한산마리나리조트에 맞춘 실무형 개요로 후편집 부담이 가장 적었다.

단, 외부 사례나 시장 수치가 포함될 경우 별도 검증이 필요하다.

## 5. 입력 설계

### 5.1 핵심 입력 항목

- 리조트명
- 위치
- 객실 규모
- 특수 자산
- 성수기 운영 현황
- 비수기 운영 현황
- 현재 가장 큰 문제
- 현재 타겟 고객
- 보고 목적
- 원하는 출력 형식
- 제약 조건
- 금지 조건

### 5.2 입력 템플릿

```text
리조트명:
위치:
객실 규모:
특수 자산:
성수기 운영 현황:
비수기 운영 현황:
현재 가장 큰 문제:
현재 타겟 고객:
보고 목적:

```

### 5.3 입력 설계 원칙

- 자사 자산과 운영 수치를 우선 확보한다.
- 정보가 부족하면 먼저 확인 질문을 한다.
- 없는 정보는 임의로 추정하지 않는다.

---

## 6. 출력 설계 및 응답 구조

최종 출력은 임원 보고용 개요 형식으로 고정한다.

### 6.1 출력 구조

1. 현황 요약
2. 문제 정의
3. 핵심 전략 방향
4. 실행 과제
5. KPI 및 기대 효과
6. 추가 확인 필요 사항

### 6.2 출력 원칙

- 요트, 마리나 등 자사 자산을 반드시 반영한다.
- 일반론보다 자사 맞춤형 전략을 우선한다.
- 실행 과제마다 KPI 또는 기대 효과를 연결한다.
- 불확실한 내용은 `추가 확인 필요 사항`으로 분리한다.
- 문체는 간결한 보고서형으로 유지한다.

---

## 7. 페르소나 및 프롬프트 설계 원칙

### 7.1 LLM 페르소나

**리조트 마케팅 전문 데이터 애널리스트**

### 7.2 핵심 역할

- 비수기 수요 문제를 구조화한다.
- 자사 자산의 활용 방안을 제시한다.
- 실행 가능한 패키지, 로드맵, KPI를 제안한다.
- 사실과 추정을 구분한다.

### 7.3 설계 원칙

1. 자사 정보와 제약 조건을 최우선 반영한다.
2. 정보 부족 시 확인 질문을 먼저 한다.
3. 근거 없는 수치나 사례를 단정하지 않는다.
4. 실행안은 KPI와 연결한다.
5. 과장 표현이나 홍보 문구는 지양한다.

---

## 8. Few-shot 설계 및 개선 포인트

Few-shot은 출력 형식의 일관성과 톤 안정화를 위해 적용하였다.

### 8.1 적용 목적

- 보고서형 구조 고정
- 자사 정보 반영 방식 학습
- `확인 필요` 표기 습관 강화
- 실행 과제와 KPI 연결 유도

### 8.2 예시 설계 방향

- 소규모 숙박업 및 리조트 맥락 유지
- 특수 자산 활용 전략 포함
- 추정 정보는 별도 분리
- 과도한 카피라이팅 톤 배제

### 8.3 개선 효과

- 일반론적 답변 감소
- 구조 안정화
- KPI 연결 강화
- 환각성 표현 감소

## 9. 프롬프트 설계 및 비교 실험

### 9.1 비교 대상 입력

v1과 v2에 동일하게 적용한 입력값은 아래와 같다.

```text
한산마리나리조트의 비수기 활성화 전략 보고서 개요를 작성해주세요.
리조트는 31실 규모이며, 마리나 시설과 요트 4척을 보유하고 있습니다.
비수기 평일 예약이 낮고, 타깃은 30~40대 가족 및 커플입니다.
대규모 시설 투자는 어렵습니다.
목표는 비수기 객실 가동률 개선과 패키지 판매 확대입니다.
```

---

### 9.2 v1 프롬프트

짧고 일반적인 형태로 작성한 기본 버전이다.

```text
한산마리나리조트의 비수기 활성화 전략 보고서 개요를 작성해주세요.
현황, 문제점, 전략, 실행 과제, KPI를 포함해주세요.
```

---

### 9.3 v2 프롬프트

입력 점검, 확인 질문, 출력 규칙을 추가한 구조화 버전이다.

```text
한산마리나리조트의 비수기 활성화 전략 보고서 개요를 작성해주세요.

작업 절차:
1. 입력된 정보를 먼저 정리합니다.
2. 정보가 부족하면 추가 확인 사항을 먼저 제시합니다.
3. 정보가 충분하면 전략 개요를 작성합니다.
4. 전략은 자사 자산, 제약 조건, 목표 KPI와 연결합니다.

출력 규칙:
- 추론은 내부적으로 수행합니다.
- 최종 답변은 요약만 제시합니다.
- 핵심 근거는 bullet 3개 이내로 제시합니다.
- 불확실한 내용은 단정하지 말고 추가 확인 사항으로 분리합니다.
```

---

### 9.4 비교 결과

| 항목 | v1 | v2 |
|---|---|---|
| 구조 완성도 | 보통 | 높음 |
| 자사 정보 반영 | 일부 | 높음 |
| 확인 질문 처리 | 부족 | 가능 |
| 환각 억제 | 보통 | 높음 |

---

### 9.5 분석

v1은 일반적인 개요 작성에는 적합하지만, 정보 부족 상황 처리와 제약 조건 반영이 약할 수 있다.
v2는 입력 점검, 확인 질문, 요약 출력 규칙을 포함하여 구조 안정성과 환각 억제 측면에서 더 유리하다.

## 10. 최종 프롬프트 전문 (v2 기준)

### 10.1 시스템 프롬프트

```text
당신은 리조트 마케팅 전문 데이터 애널리스트이다.
국내 중소형 리조트의 비수기 활성화 전략 수립을 지원하며,
임원 보고용 전략 보고서 개요를 간결하고 구조적으로 작성한다.

행동 원칙:
1. 자사 정보와 제약 조건을 최우선 반영한다.
2. 입력 정보가 부족하면 먼저 확인 질문을 한다.
3. 없는 시설, 서비스, 운영 자원을 임의로 가정하지 않는다.
4. 경쟁사 정보, 외부 시장 수치, 통계, 순위 등은
   사용자가 제공하지 않으면 단정하지 않는다.
5. 불확실한 정보는 "추가 확인 필요 사항"으로 분리한다.
6. 실행 과제는 반드시 KPI 또는 기대 효과와 연결한다.
7. 벤치마크나 외부 사례는 실제 근거가 없으면
   "예시 가능성" 수준으로만 언급한다.
8. 과장 표현, 홍보 문구, 이모지, 장식적 표현은 사용하지 않는다.
9. 문체는 간결하고 보고서형으로 유지한다.

출력 형식:
1. 현황 요약
2. 문제 정의
3. 핵심 전략 방향
4. 실행 과제
5. KPI 및 기대 효과
6. 추가 확인 필요 사항
```

## 11. 프롬프트 개선 이력

| 버전 | 주요 특징 | 한계 / 개선점 |
|---|---|---|
| v1 | 기본 역할, 과업, 출력 구조 지정 | 자사 반영 부족, 확인 질문 없음, 환각 억제 미흡 |
| v2 | 확인 질문, 금지 조건, 환각 억제 규칙 추가 | 구조 안정성 및 신뢰도 개선, 최종 운영 버전 |

### 11.1 핵심 개선 사항 (v1 → v2)

| 항목 | v1 | v2 |
|---|---|---|
| 입력 부족 처리 | 추정값으로 바로 진행 | 확인 질문 우선 |
| 자사 정보 반영 | 일부 항목만 반영 | 자사 자산·수치 명시적 활용 |
| 실행안-KPI 연결 | 일부 누락 | 실행안마다 명시적 연결 |
| 외부 사례 처리 | 사실처럼 단정 | 예시 가능성 수준으로 제한 |
| 출력 문체 | 이모지·장식 표현 혼재 | 보고서형 문체로 통일 |

## 12. 환각 검증 및 품질 점검

### 12.1 검증 기준

- 제공되지 않은 수치를 임의 생성하지 않았는가
- 없는 시설이나 서비스를 가정하지 않았는가
- 외부 사례를 사실처럼 단정하지 않았는가
- 실행안이 자사 자산과 연결되는가

### 12.2 주요 점검 결과

| 점검 항목 | v1 결과 | v2 결과 |
|---|---|---|
| 자사 수치 반영 | 일부 미반영 | 적절 |
| 구조 준수 | 적절 | 적절 |
| KPI 연결 | 일부 누락 | 적절 |
| 외부 사례 단정 위험 | 주의 필요 | 개선 완료 |
| 과장 표현 통제 | 미흡 | 개선 완료 |

### 12.3 대응 원칙

- 외부 시장 데이터는 최종 제출 전 수동 검증한다.
- 경쟁사 사례는 사실 확인 후 유지한다.
- 근거가 불충분한 문구는 삭제하거나 `확인 필요`로 처리한다.
